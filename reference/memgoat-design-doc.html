MemGoat: Cave Escape

🏆 WeMakeDevs × Cognee Hackathon 📅 Jun 29 – Jul 5, 2026 📌 Status: For team review

1. Concept in One Paragraph

A puzzle-driven escape room where the player controls a goat trying to find its way out of a multi-room cave. The goat has short-term memory loss — it is an LLM, and every 2 minutes its session resets and it forgets everything. The cave contains artifacts (tablets, paintings, statues) that hold clues, but no single clue reveals the answer. The exit is only reachable by connecting 3–4 clues into a chain. The player actively examines artifacts, feeds them into Cognee (the goat's external knowledge graph), then queries Cognee in natural language to surface the connected chain. The game's core demo moment: the timer hits zero, the goat wakes up confused, the player asks "What do I know?" — and Cognee returns the full graph of everything discovered so far. The LLM forgot. Cognee didn't.

Why this is a good Cognee fit

The game only works because of graph traversal. Individual clues are useless. The escape path emerges from Cognee connecting them. A vector database or a simple array cannot do this — Cognee's entity-relationship graph is non-negotiable for the core mechanic.

3. Player Active Actions

The player has exactly three verbs. Each maps to one Cognee API. The game is designed so all three are exercised and each feels meaningful.

Action 1

Examine

→ remember()

Player moves goat near an artifact and clicks "Examine." The LLM narrates what the goat perceives. The clue is ingested into Cognee, which extracts entities and relationships and adds them to the knowledge graph.

⚠ Limit: 3 per 2-minute session. Choose wisely.
Action 2

Ask the Cave

→ recall()

Player types a natural language question ("What connects the moon to the exit?"). Cognee performs a graph traversal and returns a connected chain — not individual clues, but a path. The chain is rendered visually as linked nodes.

💡 Skill: Better questions surface clearer chains.
Action 3

Dismiss

→ forget()

After examining a clue the player suspects is a red herring, they can dismiss it permanently. The node is removed from Cognee's graph. Irreversible — if you dismiss a real clue, it's gone until found and re-examined.

🚨 Irreversible. Adds real stakes to the decision.

The 3-Examine Limit is the Core Design Constraint

Each room contains more artifacts than the player can examine in a single 2-minute session. This forces the player to read the room, prioritize, and make bets on which clues matter. Over multiple sessions, the graph fills up incrementally. This means:

  • Players who examine strategically build a better graph faster
  • Players who examine randomly may waste sessions on red herrings
  • The constraint is a natural pacing mechanism — the game can't be rushed
Open item

Is 3 the right per-session limit? Needs playtesting. Too few = frustrating; too many = no meaningful choice. Could make it room-dependent (tutorial room has 4 slots, harder rooms have 2).

4. Game Loop

The Timer Reset is a Feature, Not a Penalty

When the LLM session resets, the player doesn't lose anything in Cognee. What they lose is conversational context — the goat forgets what it was just thinking about. The re-orientation query ("What do I know?") is the game's best demo moment: Cognee returns a structured summary of all discovered knowledge, organized as a graph, regardless of how many sessions have passed.

6. Puzzle Design Rules

These rules exist to ensure Cognee's graph traversal is genuinely necessary — not decorative.

  1. No single clue is the answer. The escape path must require connecting at least 3 artifacts. If a player can escape by reading one inscription, Cognee added nothing.
  2. Chains must flow through entities, not just keywords. Cognee extracts entities (Moon, Waterfall, Door) and relationships (triggers, reveals, opens). Puzzle clues must be written so these relationships are extractable from natural language. Avoid clues that are too abstract.
  3. Every room has at least one red herring. Red herrings must be credible enough to waste an examination slot. Good red herrings connect to real entities but lead to dead ends (e.g., "Wolf points at moonlight" — Moon is real, but the wolf path is a dead end).
  4. Recall queries must be guessable. Players should be able to intuit what question to ask. The game must give enough visual hints ("try asking about what connects things" or showing example queries). The puzzle should never be impossible to query out of.
  5. Rooms unlock progressively. Later rooms require chains from earlier rooms to unlock, creating dependency and replay of earlier knowledge.

9. Open Items & Decisions Needed

These need to be resolved before or early in the build. Ranked by urgency.

1
What do we call the Cognee layer in-game?

It should not be called "Cognee" in the game. Options: "The Cave Echo," "Ancient Memory," "Deep Instinct," "The Stone Mind." Affects all UI copy, narration, and the README story. Decide early — it propagates everywhere.

2
How many rooms, and what are the full puzzle chains?

We need room specs before building: each room's artifacts, which are real clues, which are red herrings, and what the escape chain is. Suggested: Room 1 (tutorial, 2-node chain), Room 2 (3-node chain), Room 3 (4-node chain with one active red herring to dismiss). Must be authored with care so Cognee extracts the right entities.

3
Cognee Cloud vs. self-hosted?

Cloud (free dev plan with code COGNEE-35) is faster to set up. Self-hosted scores on the "Best Use of Open Source" prize track. Decide based on which prize track we're targeting.

4
Cave UI style: top-down 2D or text-adventure?

Top-down 2D is more visually impressive for demos but requires more frontend work. Text-adventure is buildable in a day and lets us focus on the Cognee layer. Given the hackathon timeline (7 days), text-adventure is lower risk.

5
How does recall() return data to the frontend?

Cognee's recall() returns structured graph data (nodes + edges) or prose? We need to decide if we render the chain as a visual node graph (more impressive, harder) or as a formatted text answer (simpler). Chain visualization is the better demo but requires a graph rendering library (e.g., D3 or React Flow).

6
What happens if the player never uses dismiss()?

If red herrings stay in the graph, recall() might return polluted chains. Is that intentional friction? Or do we need to design puzzles where red herrings don't block the escape chain? Test this during early integration.

7
Examine slot count per session (currently: 3). Is that right?

Needs playtesting. 3 is a guess. If the goat needs to cross rooms to reach artifacts, 3 might be too few to make meaningful progress per session. Could also vary by room difficulty.

8
Multi-player / multi-instance isolation

Cognee's graph must be scoped per game session. Cognee supports multi-user mode with user_id scoping. This must be implemented from day one — retrofitting is painful.
